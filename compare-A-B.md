# Plan A vs Plan B 方案对比

> 研究组知识库管理方案选型指南

---

## 快速对比

| 维度 | Plan A: Obsidian + Git + 飞书 | Plan B: WorkBuddy + IMA |
|------|------------------------------|------------------------|
| **核心工具** | Obsidian + Git + 飞书 | IMA + WorkBuddy |
| **技术门槛** | ⭐⭐⭐ 高（需掌握 Git、Markdown） | ⭐ 低（仅需安装 IMA） |
| **部署时间** | 2-3 小时 | 30 分钟 |
| **成员上手** | 30-60 分钟 | 5 分钟 |
| **自动化能力** | ✅ GitHub Actions | ✅ WorkBuddy 自动化 |
| **本地知识库** | ✅ Obsidian 本地库 | ✅ IMA 个人库 |
| **共享知识库** | ✅ Git 仓库 | ✅ IMA 共享库 |
| **定时推送** | ✅ 飞书群通知 | ✅ 保存到 IMA（组会查看） |
| **Lint 质检** | ✅ lint_wiki.py | ✅ WorkBuddy Lint |
| **离线使用** | ✅ 完全支持 | ⚠️ 需联网（IMA 云端） |
| **数据可控性** | ✅ 完全自主（Git 仓库） | ⚠️ 依赖 IMA 服务 |
| **扩展性** | ✅ 极高（插件生态丰富） | ⚠️ 受限于 IMA API |
| **成本** | 免费 | 免费（IMA 30GB） |

## 详细对比

### 1. 成员使用体验

#### Plan A
```
成员流程：
1. 安装 Obsidian（下载+配置）
2. 安装 Git（命令行工具）
3. 克隆 Git 仓库（需理解 git clone）
4. 配置 Obsidian 插件（Git 插件、模板插件）
5. 学习 Markdown 格式规范
6. 日常使用：编辑 → git add → git commit → git push
```

**优点**：
- 完全掌控数据（本地 Markdown 文件）
- 版本历史完整（Git 记录）
- 可离线使用

**缺点**：
- 学习曲线陡峭
- 操作步骤繁琐
- 容易忘记提交/推送
- 冲突解决需要技术能力

#### Plan B
```
成员流程：
1. 安装 IMA（下载安装）
2. 加入共享知识库（扫码/邀请链接）
3. 日常使用：拖拽论文 → AI 问答 → 整理笔记 → 添加到共享库
```

**优点**：
- 开箱即用
- 内置 AI 辅助
- 无需理解 Git 概念
- 自动同步

**缺点**：
- 数据存储在 IMA 云端
- 需保持联网
- 导出格式受限

---

### 2. 管理员部署体验

#### Plan A
```
部署步骤：
1. 创建 GitHub 仓库
2. 配置 GitHub Actions（YAML 文件）
3. 配置飞书机器人（Webhook）
4. 编写 lint_wiki.py 脚本
5. 编写 CLAUDE.md 工作流定义
6. 测试自动化流程
7. 编写成员使用文档
8. 组织培训会议
```

**耗时**：2-3 小时

#### Plan B
```
部署步骤：
1. 创建 IMA 共享知识库
2. 邀请成员加入
3. 在 WorkBuddy 中配置自动化（3 个任务）
4. 测试自动化流程
5. 分发成员快速上手指南
```

**耗时**：30 分钟

---

### 3. 自动化能力对比

| 功能 | Plan A | Plan B |
|------|--------|--------|
| **定时触发** | ✅ GitHub Actions cron | ✅ WorkBuddy 自动化 |
| **Lint 检查** | ✅ Python 脚本 | ✅ WorkBuddy 内置 |
| **议题生成** | ✅ LLM API 调用 | ✅ WorkBuddy + LLM |
| **消息推送** | ✅ 飞书 Webhook | ✅ 保存到 IMA / 可选微信 |
| **自定义逻辑** | ✅ 完全可编程 | ⚠️ 受限于 WorkBuddy |
| **日志查看** | ✅ GitHub Actions 日志 | ✅ WorkBuddy 执行记录 |

---

### 4. 数据安全与可控性

#### Plan A
- **数据存储**：Git 仓库（GitHub/GitLab/自建）
- **数据格式**：纯文本 Markdown
- **备份策略**：Git 天然多副本
- **导出**：随时可导出为任意格式
- **隐私**：完全自主控制

#### Plan B
- **数据存储**：IMA 云端
- **数据格式**：IMA 私有格式（可导出 Markdown）
- **备份策略**：依赖 IMA 服务
- **导出**：支持批量导出
- **隐私**：需信任 IMA 服务商

---

### 5. 适用场景

#### 选择 Plan A 如果：
- [ ] 团队成员有较强的技术背景
- [ ] 对数据可控性要求极高
- [ ] 需要复杂的自定义工作流
- [ ] 希望完全离线使用
- [ ] 已有 Git/GitHub 使用经验
- [ ] 愿意投入时间进行培训

#### 选择 Plan B 如果：
- [ ] 追求"开箱即用"
- [ ] 团队成员技术背景参差不齐
- [ ] 希望快速落地，减少培训成本
- [ ] 接受云端存储方案
- [ ] 需要内置 AI 辅助功能
- [ ] 没有专门的运维人员

## 我们的建议

### 对于大多数研究组：推荐 Plan B

理由：
1. **降低门槛**：让非技术成员也能轻松参与
2. **快速启动**：30 分钟即可完成部署
3. **AI 原生**：IMA 内置 AI 辅助，符合当前趋势
4. **维护简单**：无需担心 Git 冲突、版本管理

### 对于技术型团队：可选 Plan A

理由：
1. **完全可控**：数据、流程、工具链完全自主
2. **高度定制**：可根据需求深度定制
3. **技术积累**：有助于团队技术能力建设

## 混合方案（进阶）

如果两个方案都想保留优点，可以考虑：

```
Plan B + 定期导出：
├── 日常使用：IMA（Plan B）
├── 组会讨论：IMA 共享库
├── 每月导出：从 IMA 导出 Markdown
└── 长期归档：存入 Git 仓库（Plan A）
```

这样既保证了日常使用的便捷性，又保留了长期数据的可控性。
